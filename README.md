# 🏦 Parabank Test Automation Framework

A professional end-to-end test automation framework for the [ParaBank](https://parabank.parasoft.com) demo banking application, built with **Playwright** and **TypeScript**. Covers UI, API, and combined UI+API test scenarios.

---

## 🚀 Tech Stack

| Tool | Purpose |
|------|---------|
| [Playwright](https://playwright.dev) | Browser automation & API testing |
| [TypeScript](https://www.typescriptlang.org) | Type-safe test development |
| [Allure](https://allurereport.org) | Test reporting |
| [dotenv](https://github.com/motdotla/dotenv) | Environment variable management |
| [GitHub Actions](https://github.com/features/actions) | CI/CD pipeline |

## ⚙️ Setup & Installation

### Prerequisites
- Node.js >= 18
- Java (required for Allure CLI)

### Install dependencies
```bash
npm install
npx playwright install --with-deps chromium
```

### Configure environment variables
```bash
cp .env.example .env
```

Edit `.env`:
```
BASE_URL=https://parabank.parasoft.com
VALID_USERNAME=your_username
VALID_PASSWORD=your_password
INVALID_USERNAME=wrong_user
INVALID_PASSWORD=wrong_pass
```

## ▶️ Running Tests

```bash
# Run all tests
npx playwright test

# Run UI tests only
npx playwright test tests/ui/

# Run API tests only
npx playwright test tests/api/

# Run combined tests only
npx playwright test tests/combined/

# Run a specific test file
npx playwright test tests/ui/login.spec.ts

# Run in headed mode (visible browser)
npx playwright test --headed

# Run with Playwright UI mode
npx playwright test --ui
```

## 🏗️ Architecture

### Page Object Model (POM)
All pages extend `BasePage`, which provides reusable low-level helpers (`BasePageClick`, `BasePageFill`, `BasePageExpectVisible`, etc.).

`ManagePage` acts as a central hub using **lazy getters** — page objects are instantiated only on first access, saving resources in large test suites.

```
BasePage (abstract)
    ├── LoginPage
    ├── RegisterPage
    ├── OverviewPage
    ├── OpenAccountPage
    ├── BillPayPage
    ├── TransferPage
    ├── RequestLoanPage
    └── FindTransactionPage

ManagePage → aggregates all pages via lazy getters
```

### Custom Fixtures
`pom.fixture.ts` extends Playwright's base `test` with:
- `pm` — `PomManager` instance with ad-blocking enabled
- `billPayApi` — `BillPayApi` instance
- `accountsApi` — `AccountsApi` instance
- `validUser` / `invalidUser` — credential fixtures from `.env`

### Authentication Strategy
Each test suite uses `test.use({ storageState: { cookies: [], origins: [] } })` to start with a **fresh session**, then registers a new user dynamically in `beforeEach`. This ensures full test isolation without relying on pre-existing data.

## 🚫 Known Limitations

These are known quirks of the ParaBank demo site, not framework bugs:

| Issue | Details |
|-------|---------|
| `POST /billpay` — no body validation | Sending empty or missing payee data still returns 200. Validation is UI-only. |
| `POST /billpay` — invalid `accountId` | Returns HTTP 500 with an HTML error page instead of JSON. Always check `response.status()` before calling `response.json()`. |
| Site instability | ParaBank is a free public demo site and may be slow or temporarily unreachable. `navigationTimeout` is set to 60s to handle this. |
| Usernames must be unique | Each test registers a new user with a timestamp-based username (`amine{timestamp}`) to avoid conflicts. |

```yaml
# Triggered on: push, pull_request, workflow_dispatch
# Browser: Chromium
# Retries: 2 (CI) / 1 (local)
```

Artifacts uploaded per run:
- Playwright HTML report
- Allure results
- Screenshots & videos on failure
